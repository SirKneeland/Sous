## Conventions

- **Error handling:** `Result<Success, Failure>` for synchronous operations (e.g., PatchValidationResult, DecodeResult); `throws` with typed Error enums for orchestration (e.g., PatchApplierError). Large discriminated `LLMError` enum covers all network/decode/validation failure cases.
- **Async:** async/await throughout. Task.isCancelled checked after every await. Generation counter (Int) in AppStore prevents stale-task mutations after cancellation races.
- **Concurrency:** All model types are Sendable (structs/enums with Sendable properties). AppStore is @MainActor. LLMClient and LLMOrchestrator protocols are Sendable.
- **Naming:** Protocols use `-Client`, `-Orchestrator`, `-Preparator` suffixes. Error types use `-Error` or `-Failure` suffixes. Result types use `-Result`. DTOs use `-DTO`. Test mocks use `Mock-` prefix.
- **Type safety:** Heavy discriminated enum use (UIState, Patch, LLMResult, StepStatus, etc.) ‚Äî minimal Optional usage.
- **JSON:** Two-pass decoding (strict parse ‚Üí brace-extraction fallback) to handle malformed LLM output. Manual encoding (no Codable), fully explicit.
- **Debug guards:** `#if DEBUG` around logging and environment variable fallback. Prompts, recipe text, and API keys are never logged.
- **Testing:** All core state logic has unit tests. Mocks injected via protocol (no swizzling). Tests are behavioral, not implementation-coupled.
